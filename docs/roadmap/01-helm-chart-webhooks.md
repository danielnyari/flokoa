# 01 — Helm Chart: Webhooks, Network Policies, Completeness

**Phase:** 0 · **Size:** M · **Depends on:** — · **Enables:** 02 (chart publish), 05/06 (validation + policies in the install path)

## Goal

Make `helm install flokoa` produce a complete, secure-by-default installation: all CRDs, admission webhooks enabled via cert-manager, and network policies — so CRD validation (which units 04/05/07 extend) actually runs in real deployments.

## Current state

- Chart at `operator/charts/flokoa/` (`version: 0.1.0`, `appVersion: "0.1.0"`); values top-level keys: `images`, `crds`, `controller`, `server`, `dex`, `agentWorkflow`, `argo`, `extraObjects`. Template dirs: `argo/`, `controller/`, `dex/`, `server/`, plus `crds.yaml`, `extra-manifests.yaml`, `NOTES.txt`, `_helpers.tpl`.
- `files/crds/` contains all 7 CRD YAMLs **including** `agent.flokoa.ai_agentworkflows.yaml` — the `RELEASE_REVIEW.md` §7.2 P0 ("AgentWorkflow CRD missing") appears already fixed; **verify and close it out** rather than re-implementing.
- **No webhook templates exist.** Webhooks are fully implemented in code — 6 validators in `operator/api/v1alpha1/*_webhook.go` (e.g. `SetupAgentToolWebhookWithManager` registering `AgentToolCustomValidator` via `ctrl.NewWebhookManagedBy`) plus `internal/webhook/v1alpha1/agentworkflow_webhook.go` — and kubebuilder markers declare paths like `/validate-agent-flokoa-ai-v1alpha1-agenttool`. Kustomize manifests for the webhook service/configurations live under `operator/config/webhook/` (generated by `make manifests`), but the chart never renders them, so production installs run **without admission validation**.
- NetworkPolicies exist only as kustomize samples: `operator/config/network-policy/allow-metrics-traffic.yaml` (port 8443 from namespaces labeled `metrics: enabled`) and `allow-webhook-traffic.yaml` (port 443 from namespaces labeled `webhook: enabled`). Not in the chart.

## Target design

- `webhooks.enabled` value (default **true**) renders: webhook `Service`, `ValidatingWebhookConfiguration` (mirroring `operator/config/webhook/manifests.yaml`, which is the generated source of truth for paths/rules), cert-manager `Certificate` + `Issuer` (self-signed) with `cert-manager.io/inject-ca-from` annotations, and the cert volume mount on the controller Deployment.
- `webhooks.certManager.enabled=true` (default) uses cert-manager; document it as an install prerequisite. No bespoke cert generation path in v1 — `helm template`-time `genCA` breaks on upgrades; rejected.
- `networkPolicies.enabled` value (default false, flip later) rendering deny-by-default + allow rules for: webhook ingress (from kube-apiserver), metrics scrape, server ingress, dex.
- Chart CI: `helm lint` + `helm template` golden-file test + (stretch) `chart-testing` install against Kind in `test-e2e.yml`.

## Implementation plan

1. **Verify current chart state** — confirm `files/crds/agentworkflows` is current with `operator/config/crd/bases/`; if `make manifests` output drifts from `files/crds/`, add a Makefile target `sync-chart-crds` that copies generated CRDs into the chart and call it from `manifests`. This kills an entire class of drift (the original P0) permanently.
2. **Find the webhook gate in `operator/cmd/main.go`** — locate how webhook setup is gated (flag or `ENABLE_WEBHOOKS` env; RELEASE_REVIEW references `--enable-webhooks`). The chart must set this gate when `webhooks.enabled`.
3. **Create `templates/controller/webhook.yaml`** rendering Service (port 443 → 9443), ValidatingWebhookConfiguration with one entry per resource matching the kubebuilder markers (agents, agenttools, models, modelproviders, instructions, agenttriggers, agentworkflows). Generate the rule list from `operator/config/webhook/manifests.yaml` — do not hand-write paths.
4. **Create `templates/controller/certificate.yaml`** (Issuer + Certificate, `{{ if .Values.webhooks.certManager.enabled }}`), wire `caBundle` injection annotation, mount the secret at the path controller-runtime expects (`/tmp/k8s-webhook-server/serving-certs` default).
5. **Patch `templates/controller/deployment.yaml`**: webhook port, cert volume, the gate flag/env from step 2.
6. **Create `templates/network-policies.yaml`** porting the two kustomize policies + default-deny, gated on `networkPolicies.enabled`.
7. **Values + docs**: add `webhooks:` and `networkPolicies:` blocks to `values.yaml` with comments; update `NOTES.txt`; document cert-manager prerequisite in chart README.
8. **CI**: add `helm lint operator/charts/flokoa` and a `helm template … | kubectl apply --dry-run=server` (or golden snapshot diff) step to `.github/workflows/test.yml`.

## Testing

- `helm template` snapshot tests: default values, `webhooks.enabled=false`, `networkPolicies.enabled=true` — assert webhook config paths match the kubebuilder marker paths.
- E2E (existing Kind setup, `make setup-test-e2e`): install chart with webhooks enabled + cert-manager, apply an invalid `AgentTool` (e.g. `type: openapi` without `openApi` block), assert admission rejection. Add as a Ginkgo case under `operator/test/e2e/`.

## Acceptance criteria

- Fresh Kind cluster: `helm install flokoa ./operator/charts/flokoa` (with cert-manager preinstalled) yields a Ready operator, all 7 CRDs, working validation webhooks (invalid CR rejected with the validator's message).
- `webhooks.enabled=false` install still works (current behavior preserved).
- `make manifests` keeps chart CRDs in sync (drift impossible or CI-detected).
- RELEASE_REVIEW.md §7.2 marked resolved with evidence.

## Out of scope

- Publishing the chart (unit 02). Mutating/defaulting webhooks (none exist today). Cert rotation beyond cert-manager defaults.
